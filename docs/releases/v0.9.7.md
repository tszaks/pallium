# v0.9.7

Pallium v0.9.7 makes workflows adopt the model of whatever agent is steering
them, ships a built-in Claude provider, and fixes a prompt-size crash and a
recurring test flake.

## Added

- Worker provider now adopts the steering agent by default: running a workflow
  from inside Claude Code uses Claude workers automatically, with no per-agent
  `provider:` and no wrapper. Precedence is per-agent `provider` >
  `opts.Provider` > `PALLIUM_WORKFLOW_PROVIDER` > detected steering agent
  (`CLAUDECODE` -> `claude`) > `codex` fallback.
- Built-in `claude` provider that invokes the `claude` CLI directly when the
  resolved provider is `claude` and no wrapper command is configured, so
  adoption works with only the CLI on PATH. The prompt is piped over stdin, and
  the worker runs with `--safe-mode --strict-mcp-config` so ambient repo or user
  settings, hooks, and MCP servers cannot widen its tool allowlist.
- Opt-in worker network access, safe by default: an agent only gets egress when
  it sets `network: true` AND the run is started with `--allow-network`, exposed
  consistently on the CLI, MCP, and HTTP entry points.
- `PALLIUM_WORKFLOW_CLAUDE_ALLOWED_TOOLS` to extend the built-in provider's edit
  allowlist for non-Go repos; the default edit allowlist now spans Go, Node,
  Deno, Python, Rust, Make, and Apple/Swift build and test runners.

## Fixed

- Passed provider prompts to worker subprocesses only through a file, never
  inline in the environment, so a large prompt no longer overflows the exec
  argument limit (`argument list too long`).
- Drained the console PTY before closing it, so a fast child's output is no
  longer discarded (the recurring `TestConsoleRunReadAndOwnedList` flake).
- Surfaced a `claude` CLI error envelope (`is_error`) as an error instead of
  returning it as the answer, and capped CLI output embedded in errors.
- Reporting and analytics no longer re-run steering detection on historical
  runs, so a past `codex` run is not relabeled `claude` when inspected later.
