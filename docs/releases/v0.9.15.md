# v0.9.15

The "operational honesty" batch: a durable fix for the stale run/agent registry an outside
Ultra-vs-Pallium audit flagged on 2026-07-12, a self-installing adoption trigger, and forensics on
that audit's own claims.

## Fixed

- **Stale run/agent auto-reconcile.** A killed `pallium workflow run` process used to leave its run
  and agents looking "running" forever — the audit found 3 runs and 16 agents stuck this way, some
  from days earlier. Every run now records its owning process id and a heartbeat
  (`ClaimRunOwnership`/`HeartbeatRun`); `workflow list/status/inspect/fleet/analytics` reconcile on
  every call — a "running" row whose owner process is confirmed dead (or, for a pre-0.9.15 row that
  was never claimed, whose heartbeat predates a time window) flips to a new honest `interrupted`
  status, and any agent whose parent run is no longer running is swept the same way. Reconciled-not-
  hidden: list/status/fleet/analytics print exactly what just changed. `workflow gc --stale
  [--stale-after-minutes n]` runs the same sweep as an explicit bulk operation. Verified live against
  the real registry: the audit's exact 3 runs / 16 agents are now `interrupted`, not `running`.
- **Not-done is now unmissable.** `workflow status`/`report` print a blunt `VERDICT: FINISHED` /
  `VERDICT: NOT FINISHED (<status>) — N/M agents complete` line first, closing the exact ambiguity
  that let an outside audit poll "no agents completed yet" repeatedly and then still declare its
  verification clean.
- **`workflow validate` catches the validate-vs-run gap for `.then()`/`.catch()` chained on
  `agent()`/`check()`.** This compiled clean and only broke at `run` time (see the existing
  `stageChainHint` runtime message) — validate now emits a non-fatal warning for the same pattern
  (`workflow generate` surfaces it too), so the mismatch between what validate approves and what run
  can actually execute is smaller.

## Added

- **The adoption trigger installs itself.** `pallium agents install` has existed since PR #14 but
  went unrun on a real machine for 4 days because nothing ever prompted for it. `pallium start` now
  offers (never silently writes) to install the AGENTS.md/CLAUDE.md block when a repo doesn't have
  one yet — a plain `[y/N]` prompt, skipped entirely in `--json` mode, non-interactive stdin, or with
  `PALLIUM_SKIP_ADOPTION_HINT` set. `pallium version` prints a one-line hint under the same
  suppression rule when the current directory has no block.

## Forensics: the 2026-07-12 Ultra-vs-Pallium audit

- Its three background verification tasks (`pallium_architecture`, `ultra_baseline`,
  `pallium_smoke`) are **not Pallium workflow runs** — no matching row exists anywhere in the
  98-run registry. They were Codex's own native background-task feature (the same "Ultra" subagent
  mechanism the audit was comparing against), and that specific Codex session was never indexed by
  Pallium's session store either (nothing indexed after 2026-07-08). There is no artifact on this
  machine to confirm or refute its "13 tests passed / build+vet+race clean" claim — UNVERIFIABLE,
  not confirmed. The out.txt/schema.json/.playwright-mcp debris the audit's own `git status` showed
  predates it by 5 days (2026-07-07) and is unrelated; its "no repository files were changed" claim
  holds.
- Re-based its claimed gaps against 0.9.14 (it audited 0.9.13): **already fixed** — failure
  isolation (`parallel(arrayOfThunks)` crash isolation), background resume crash, test-DB leakage
  (all shipped in 0.9.14, confirmed against source and that release's own notes). **Not a bug,
  already documented** — codex cost/usage tracking: `codex exec` emits no usage envelope at all,
  true of the regular worker path too (see PALLIUM.md, team/loop status output). **M2-scope,
  confirmed still unbuilt, no action this batch** — per-agent steer/stop/restart and
  workflow-to-team composition (teams M2, deliberately excluded from this release).

## Also fixed (found blocking this release's own CI)

- **Date-dependent test flake in `internal/sessionmemory`.** `TestRelatedRanksRepoAndFileMatches`
  hardcoded a 2026-06-10 timestamp; `recencyScore`'s 30-day cutoff (search.go) means that date
  aged past the cutoff as real time passed, dropping the non-matching "other" session's score to
  exactly 0 and filtering it out of the results entirely — degenerating the test's two-result
  comparison into comparing "target" against itself. Confirmed pre-existing on a clean `main`
  checkout (`git stash` + rerun, same failure). Fixed by using a timestamp relative to
  `time.Now()` instead of a fixed calendar date.
