# v0.9.11

Pallium v0.9.11 adds Loops: a bounded, named cycle you advance one tick at a time — the
third engine pillar alongside workflows and agent teams. Also ships hardening for Agent
Teams M1 and a safety net for the shared local database.

## Added

- `pallium loop start|tick|status|list|stop|reset` — a persistent, named cycle. `loop
  start <name> --script f.js` registers it against a script that declares
  `meta.kind === "loop"`; `loop tick <name>` advances it by exactly one bounded round —
  no daemon, cron/a trigger/an agent decides when to tick again, spawning a FRESH child
  workflow run each time (a loop never reuses one run across ticks, so there is no
  cross-tick cache collision and no risk of hitting a run's lifetime agent cap purely
  from ticking).
- Terminal-state vocabulary (`success`/`no_op`/`blocked`/`exhausted`/`stagnated`/
  `already_running`) maps to distinct process exit codes, so a calling cron/agent can
  branch without parsing JSON.
- Stagnation detection via a script-authored `signature` contract (a script opts in by
  returning `{ state, signature }`; a repeated signature across cycles trips the
  configured `stagnationThreshold`). Error ticks never advance or reset the counter.
- Budget enforcement (`cycleBudgetUsd`/`lifetimeBudgetUsd` in a loop's `meta.loop` block)
  overrides a script's self-reported state to `exhausted` regardless of what it claims —
  the same hardening already shipped for team budgets, and the same cost-tracking
  caveat: codex reports no usage/cost at all, so a codex-only loop's ceiling cannot
  self-enforce; `loop status` says so plainly.
- Reference template: `examples/workflows/review-until-clean.js` — drives review
  findings to zero one cycle at a time.
- `ARCHITECTURE.md` — Pallium's kernel/services architecture ruling: one shared kernel
  (SQLite store, provider dispatch, worktree/patch machinery, budgets, leases, the run
  journal), distinct services on top (repo intelligence, session awareness, workflows,
  loops, agent teams, adoption layer), composition only through each service's own front
  door. `PALLIUM.md` now opens with a "What does Pallium do?" six-service map.

## Changed

- `PALLIUM_TEST_DB` (introduced for `pallium team`) now also covers `pallium loop` — set
  it once per test/dogfood session and any `team ...`/`loop ...` call that forgets `--db`
  redirects there instead of the real shared global database, with a loud stderr warning.

## Fixed (Agent Teams M1 hardening)

- A failed codex team-turn stream now surfaces the real read error instead of silently
  falling through to a misleading generic failure.
- `team run`'s reported turn count now reflects turns that actually ran, not every
  member merely offered a turn in a round.
- A team turn's prompt no longer embeds the full unbounded task-board history — capped
  to open tasks plus the 5 most recent completions with truncated results.
