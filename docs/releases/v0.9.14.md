# v0.9.14

Pallium v0.9.14 fixes three dogfood bugs surfaced by real sessions, and closes a gap that let
test/dogfood runs silently write into the real global database.

## Fixed

- **`parallel(arrayOfThunks)` crash isolation**: the no-mapper thunks form of `parallel()` — the
  shape Ultracode-native agent reflexes reach for by default — was accepted at validate time but
  did not get the same crash isolation as the documented `parallel(items, fn)` form. One transient
  agent error inside a thunk used to abort the entire run instead of degrading that item to `null`.
  Both forms now share the same fatal-vs-transient classification, capture rollback, and null-drop
  behavior, so an ordinary failure in one thunk no longer takes down the rest of the batch.
- **`workflow resume <id> --background` crash**: background workflow launches derive a
  deterministic owned-session id (`"workflow-" + run.ID`), so resuming the same run in the
  background a second time reused the same `owned_sessions.id`. A plain `INSERT` hit that primary
  key on the legitimate re-entry and surfaced a raw `UNIQUE constraint failed` error instead of
  starting the resumed run — foreground resume never went through this path, so only
  `--background` crashed. `CreateOwnedSession` now upserts, resetting the row to the new launch's
  values.
- **`PALLIUM_TEST_DB` did not cover `workflow`/`console` commands**: the env var was documented as
  covering every `team`/`loop` subcommand, but `workflow`/`console` opened their stores directly
  instead of through the shared chokepoint, so it silently did nothing for them. Any dogfood or
  test run without an explicit `--db` landed in the real global database
  (`~/.pallium/codex-sessions.sqlite`) regardless of the env var — the same failure mode that once
  required a manual cleanup for `team`/`loop`, and one that leaked real rows into production during
  this release's own testing. All `workflow.Open`/`console.Open` call sites now route through
  `resolvePalliumDBPath`, matching the existing `team`/`loop` pattern.

## Verification

```bash
node npm/scripts/test.js
go vet ./...
go test ./... -count=1 -race
bash scripts/workflow-verify.sh
pallium workflow audit --run-acceptance
```

Every fix has a regression test verified by temporarily reverting the fix, confirming the test
fails, then restoring it. The `parallel(arrayOfThunks)` and `workflow resume --background` fixes
were also dogfooded end to end through the real CLI, not just as unit tests.

## Install

```bash
npm install -g pallium
pallium version
```
