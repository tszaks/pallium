# v0.9.16

Agent Teams M2, the composability milestone — complete. `team.*` primitives inside
workflow/loop scripts, a real plan-approval loop, quality-gate hooks, individual teammate
supervision (stop/restart/steer), and a fix for the M1 review's deferred zombie-decision gap.
Two PRs, six-plus rounds of adversarial bot review between them, ~35 real findings fixed
along the way — nearly all in the gate/timeout/scheduling interactions this milestone
introduced, not cosmetic. A handful were deliberately declined and documented as backlog
rather than rushed.

## Added

- **`team.*` primitives in workflow/loop scripts.** A workflow — or a loop tick — can now
  convene and drive an Agent Team programmatically: `team.create/spawn/send/approve/reject/
  gate/status/wait/stop` and `team.tasks.create/list/claim/complete`. Every primitive calls
  the exact same Store methods (or the exact same `RunTeam`/`CreateTeamTaskWithGate`/
  `CompleteTaskWithGate` runtime helpers) `pallium team ...` uses — front-door composition
  per the kernel/services architecture ruling. `team.create()` is idempotent across a
  paused/resumed workflow (a deterministic id keyed on the run and a per-run call counter,
  mirroring how `agent()`/`gate()` calls get their own replay keys) — a resumed script no
  longer creates a duplicate team.
- **Plan-approval flow.** `pallium team spawn --plan-required` (or `team.spawn(...,
  {planRequired: true})`) spawns a teammate forced read-only with `plan_status: "pending"`.
  It submits its plan as an ordinary message to `lead`. `team approve`/`team.approve()`
  validates a plan is actually pending, flips to edit mode, and journals the approval.
  `team reject <feedback>`/`team.reject()` delivers feedback and leaves the member exactly
  where it was — rejection isn't terminal here, only approval is.
- **Quality-gate hooks.** `team gate set <team-id> --hooks task_created,task_completed,
  teammate_idle "<prompt>"` (or `team.gate()`) configures an autonomous read-only verifier —
  same shape as the workflow `gate()` primitive, via a new run-independent dispatch
  (`runTeamGate`) since a team isn't a workflow run. Gate prompts include the team's goal and
  full task context (description, dependencies) so a verifier has real facts to check
  against, not just a bare title. `SetTeamGate` rejects a non-empty hooks list paired with an
  empty prompt (that combination was silently inert). `team gate set <id> --hooks ""` clears
  a configured gate.
- **Individual teammate supervision (`team member stop|restart|steer`, PR B).** The last
  piece of M2's original scope, closing the Ultra-audit gap on per-agent control. All three
  are SOFT — no in-flight kill: a turn already running when `stop`/`steer` lands keeps running
  to its own natural completion, and the supervision takes effect starting that member's next
  turn. A `stop_requested` column (independent of the member's own `Status`, which
  `FinishMemberTurn` writes unconditionally from whatever the turn itself decided) survives
  turn completion by construction, and is checked by `BeginMemberTurn`'s own acquisition CAS,
  not just the scheduler's earlier eligibility snapshot. `restart` clears the stop, reconciles
  only the named member's own stale turn (never a team-wide sweep that could clobber another
  member's genuinely-in-flight turn), and nudges it so it's actually re-scheduled. `steer`
  delivers a distinctly-framed high-priority mailbox message for the member's next turn.
  Workflow primitives: `team.member.stop/restart/steer(...)`.
- **Honest team cost display, structured, everywhere.** `team status --json` and the
  `team.status()` workflow primitive both carry `untracked_cost_providers` as a field —
  codex reports no usage/cost at all, same documented platform limitation as loops/regular
  workers.

## Fixed

Two PRs, both survived heavy adversarial bot review before merging — the short version:

- **Zombie decision-side-effect gap (ticket #32, deferred from M1).** Decision side effects
  (message send, claim, complete) now only ever apply after `FinishMemberTurn`'s lease check
  succeeds, not before. A stale-takeover zombie's decision mutates nothing.
- **Gate correctness, timeouts, and concurrency.** Gate verdicts are now schema-validated
  (fail closed on malformed JSON, not fail-open); a gate's own reported spend is recorded
  against team/turn budgets and can park a team on its own, not just a turn's own cost;
  `runTeamGate`'s timeout only applies when the caller has no deadline of its own (previously
  clobbered a longer `--agent-timeout`); `team.wait`, `team.tasks.create/complete`, and the
  shared `teamRunner()` closure all now honor `pallium workflow stop/pause` and pass correct
  run-id metadata to configured wrapper providers; `runTeamGate` no longer mutates the shared
  concurrent-turn `*Runner`'s `CodexBinary` field (a real data race under `RunTeam`'s
  concurrent scheduling).
- **Scheduling edge cases.** A `teammate_idle` gate rejection (or malfunction) now reliably
  reschedules the member instead of only flipping a status field the scheduler doesn't key
  off; same-turn `claim_task_id` + `complete_task_id` on one task now succeeds under a
  `task_completed` gate (it used to fail eligibility because the claim hadn't landed yet at
  gate-check time); a `task_completed` gate malfunction now fails closed and requeues the
  owner instead of stranding it idle with a stuck task.
- **A stale worktree could permanently block a team member's edit mode**, and **a killed
  turn's worktree could be deleted out from under a still-running one** (the latter an
  accepted, documented tradeoff, not fully closed — real per-turn liveness tracking for team
  members is bigger than this milestone).

## Declined, documented as backlog (not fixed this milestone)

- Team spend vs. workflow `--max-budget-usd`: intentionally separate budget systems today;
  unifying them is a real design question, not a quick fix.
- Pause mid-`team.wait` finalizes the interrupted member's turn as an ordinary provider
  failure rather than requeuing it — touches the still-unreconciled gap between workflow
  script resume and a team's own durable resume; likely folds into M3's external-attach work.
- A claude member's `--session-id` can be permanently "burned" by a failed first attempt
  (found live during PR A's kill/resume proof) — real, understood, earned its own focused
  session rather than a same-PR patch.
- Edit-mode worktree patch capture/apply is not lease-gated (found by two independent live
  review teammates during PR A) — pre-existing M1-era gap.
- A hard mid-turn kill for `team member stop`/`steer` (vs. the soft, turn-boundary semantics
  shipped) — deliberately not built; a `--force` flag is a designed-but-unbuilt door if real
  usage ever proves it necessary.

## Live proof, both PRs

**PR A**: a real 3-member `pallium team` (`correctness-reviewer`/`concurrency-reviewer`/
`zombie-hunter`) adversarially reviewed this batch's own diff and found real bugs live; a
real workflow script (`team.create` → `team.spawn` with `planRequired` → `team.wait` →
`team.approve` → `team.wait` → gate-checked completion) ran end to end against a real toy
repo; a live kill/resume proof (`kill -9` the steering process, confirm the orphaned `claude`
subprocess survives independently, `team attach` reconciles, `team run` resumes) surfaced two
of the real bugs above.

**PR B**: two real claude teammates, real spend ($0.3268), a toy repo. `worker-b` stopped
while idle — a real `team run` afterward gave it zero turns despite a genuinely claimable
task on the board. `worker-a` steered before ever taking a turn ("focus only on
CONTRIBUTING.md, ignore CODE_OF_CONDUCT.md") — its own real decision text afterward reads
*"Per lead's earlier directive, not picking up the CODE_OF_CONDUCT.md task... explicitly
deprioritized by lead,"* and the file landed on disk. `worker-b` then restarted, ran again,
and completed the second task for real.

## Verification

```bash
go build ./...
go vet ./...
go test ./... -count=1 -race
bash scripts/workflow-verify.sh
pallium workflow audit --run-acceptance
```

Every fix across both PRs has a regression test verified by temporarily reverting it and
confirming the test fails with the predicted message, then restoring it — no exceptions,
including the riskiest ones (the `FinishMemberTurn`/`stop_requested` race, the
`BeginMemberTurn` acquisition CAS, delivery-order/durability fixes). State-ownership and
universality lints pass unchanged — this milestone only added columns to already-covered
tables, no new tables.

One known flaky test, timing-sensitive under system load (not a regression):
`TestTeamTasksCreatePrimitiveHonorsWorkflowStop` waits 100ms before asserting a background
process has started; under heavy concurrent load (e.g. running directly after a full
`-race` pass) that window can occasionally be too tight. Passes reliably in isolation and on
every CI run for this milestone; same class as the already-documented PTY-drain and
concurrency-timing flakes elsewhere in this codebase.
