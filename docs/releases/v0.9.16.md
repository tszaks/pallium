# v0.9.16

Agent Teams M2, the composability milestone: `team.*` primitives inside workflow/loop scripts,
a real plan-approval loop, quality-gate hooks, and a fix for the M1 review's deferred
zombie-decision gap. Plus three real bugs found by dogfooding this batch on itself with a live
`pallium team` review and a live kill/resume proof — two fixed, one documented for a focused
follow-up session.

## Added

- **`team.*` primitives in workflow/loop scripts.** A workflow — or a loop tick — can now convene
  and drive an Agent Team programmatically: `team.create/spawn/send/approve/reject/gate/status/
  wait/stop` and `team.tasks.create/list/claim/complete`. Every primitive calls the exact same
  Store methods (or the exact same `RunTeam`/`CreateTeamTaskWithGate`/`CompleteTaskWithGate`
  runtime helpers) `pallium team ...` uses — front-door composition per the kernel/services
  architecture ruling, nothing reaches into team internals a CLI caller couldn't also reach. A
  fresh helper `Runner` drives any call that dispatches real provider turns (`wait`) or a gate
  check (`tasks.create`/`tasks.complete`) — never the workflow's own `Runner`, whose `Run.ID` is
  load-bearing for every `agent()`/`check()` call already in the script.
- **Plan-approval flow.** `pallium team spawn --plan-required` (or `team.spawn(..., {planRequired:
  true})`) spawns a teammate forced read-only with `plan_status: "pending"`. It submits its plan
  as an ordinary message to `lead` — a real, durable mailbox artifact, not a new schema field.
  `team approve`/`team.approve()` validates a plan is actually pending, flips to edit mode, and
  journals the approval as a message. `team reject <feedback>`/`team.reject()` delivers feedback
  and leaves the member exactly where it was — rejection isn't terminal here, only approval is;
  the member revises and resubmits next turn. A plain (non-plan-required) member's `approve` is
  unchanged: still the simple M1 mode-flip.
- **Quality-gate hooks.** `team gate set <team-id> --hooks task_created,task_completed,
  teammate_idle "<prompt>"` (or `team.gate()`) configures an autonomous read-only verifier — same
  shape as the workflow `gate()` primitive, reused via a new run-independent dispatch (`runTeamGate`)
  since a team isn't a workflow run and `gate()`'s own path requires one. `task_completed`: a
  rejected completion is delivered to the owner as feedback and never marked done (checked before
  completing, not completed-then-reverted). `teammate_idle`: a rejected idle declaration is forced
  back to `active` with the gate's reason as the note. `task_created`: a rejected task lands
  `blocked`, inserted with that status from a single `INSERT` — never claimable for even an
  instant (see Fixed, this was tightened during dogfooding).
- **Honest team cost display, structured.** `team status --json` now carries
  `untracked_cost_providers` as a field, not just a line in the text renderer — codex reports no
  usage/cost at all, same documented platform limitation as loops/regular workers.

## Fixed

- **Zombie decision-side-effect gap (ticket #32, deferred from M1's third review round).**
  `SendTeamMessage`/`ClaimTask`/`CompleteTask` used to run BEFORE `FinishMemberTurn`'s lease check,
  so a turn whose lease was already stolen by a stale takeover while its provider call was still
  in flight could still mutate the mailbox and task board — the lease failure was only discovered
  afterward. Reordered: `FinishMemberTurn` (the same atomic lease-check used everywhere else) now
  runs first; every decision side effect is unreachable if it fails. A stale-takeover zombie's
  decision now mutates nothing.
- **A stale worktree could permanently block a team member's edit mode.** Found live by this
  batch's own kill/resume acceptance proof, not synthetic: a team member reuses the SAME
  deterministic worktree path on every turn (keyed on its stable member id), so killing the
  steering process mid-turn left that turn's worktree directory on disk, and the member's next
  turn's `createWorktree` call failed outright — `git worktree add` refuses to target an existing
  directory — with no automatic recovery. Pre-existing M1-era gap, not introduced by M2; fixed
  because M2's own live proof is what surfaced it. `createWorktree` now clears a stale leftover at
  its target path before creating.

## Found live, documented for a focused follow-up (not fixed this batch)

- **A claude team member's `--session-id` can be permanently "burned" by a failed first attempt.**
  Found continuing the SAME live kill/resume proof after the worktree fix: claude's CLI refuses to
  reuse a `--session-id` a second time even when its first use never established a working
  session (it errored on something unrelated before doing real work) — but Pallium's own
  `SessionEstablished` tracking only flips true on a turn that finishes non-error, so it keeps
  retrying with the same now-permanently-rejected id. This touches the exact session-lifecycle
  code M1's third review round already hardened once (the "claude first-turn permanent brick" P1
  fix) — real, understood, but this class of code has earned a focused session of its own rather
  than a same-PR patch.
- **Edit-mode worktree patch capture/apply is not lease-gated at all** (flagged independently by
  two of the three live review teammates). Unlike decision side effects (now correctly guarded
  above), a zombie edit-mode turn's `git apply --3way` still lands in the real repo — the check
  happens unconditionally on any successful provider call, before the lease is ever re-validated.
  Pre-existing M1-era gap; the file's own comments already document a related, accepted
  concurrency tradeoff, but not this specific same-member stale-takeover angle.
- **`CreateTeamTaskWithGate`'s original create-then-flip window** (fixed in this same PR after a
  live adversarial-review teammate found it — see Verification) is recorded here only as the
  "why" for the fix above; no outstanding action.

## Live dogfooding, as instructed

Two live exercises, using the real `pallium team` CLI (not simulated), ~$3.71 total real API
spend:

1. **Adversarial review of this batch's own riskiest pieces**, as an actual 3-member `pallium
   team` (`correctness-reviewer`/`concurrency-reviewer`/`zombie-hunter`, distinct lenses, real
   claude turns, real task board). Found the `CreateTeamTaskWithGate` race (fixed, see Fixed;
   originally reported as a task_created gate's create-before-check window letting a rejected
   task get claimed before it was ever blocked) and the edit-mode patch-apply gap above.
2. **UX friction, as a first-time user of `pallium team`** (M2/M3 input, not fixed here):
   - `teamDecisionSchema`'s required `"summary"` field failed validation in **6 of 6** real
     decisions across this session — every reviewer, every turn, including after a manual nudge
     and retry. A large `complete_result` reliably crowds out a required trailing field. The
     content survived (captured as raw text in `last_turn_error`), but nothing was ever actually
     applied — no message sent, no task claimed or completed.
   - **Silent stall on a malformed decision.** Once a decision fails schema validation, the member
     goes `active` with no new mail and no task-board change — the scheduler's own "don't
     re-offer an unchanged board" optimization (correct M1 behavior in isolation) then means it
     never gets another turn without a manual `team nudge`. Observed directly: round 2 ran 0
     rounds / 0 turns for exactly this reason.
   - Both are real gaps worth M3 attention: loosen the schema (make `summary` optional with a
     derived fallback, or reorder/emphasize it), and consider auto-nudging a member whose own
     decision just failed to parse instead of leaving it to stall.

## Verification

```bash
go build ./...
go vet ./...
go test ./... -count=1 -race
bash scripts/workflow-verify.sh
pallium workflow audit --run-acceptance
```

Every fix has a regression test verified by temporarily reverting the fix and confirming it fails:
`TestRunTeamTurnZombieDecisionMutatesNothing`, `TestCreateTeamTaskWithGateNeverClaimableWhileGateInFlight`
(the live-review-found race), `TestCreateWorktreeRecoversFromStaleDirectoryFromAKilledTurn` (the
live-kill-resume-found worktree gap). Full suite green, `-race` included, no exceptions. State-
ownership and universality lints pass unchanged — this batch added columns to already-covered
tables (`teams`, `team_members`), not new tables, so no lint extension was needed; verified by
inspection, not just by the lint passing.

Live proof, in addition to the dogfooding above: a real workflow script (`team.create` → `team.spawn`
with `planRequired` → `team.wait` → `team.approve` → `team.wait` → `team.gate`-checked completion)
ran end to end against a real toy repo with real claude calls — the coder submitted a plan, got
approved, added a real function and test, discovered and fixed a missing `go.mod`, ran the real
test, and the gate genuinely evaluated the real output and approved it. A separate live kill/resume
proof (background `team run`, `kill -9` the steering process, confirm the orphaned `claude`
subprocess survives independently, `team attach --stale-after-minutes 0` reconciles, `team run`
resumes) is what surfaced the worktree and session-id findings above.

## Deferred (M2 scope boundary, not scope creep)

Individual teammate supervision — steer (priority message jumps the mailbox queue), stop
(graceful, finish current turn then park), restart (fresh turn via the persisted session token) —
is a separate PR, per the explicit split agreed before this batch started. Builds on 0.9.15's
heartbeat/liveness groundwork; closes the Ultra audit's biggest supervision gap.
