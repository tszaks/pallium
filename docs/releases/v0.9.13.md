# v0.9.13

Pallium v0.9.13 closes out the two remaining fix-it items from the reliability work started in
0.9.9-0.9.12: `gate()`/`check()`/`verify.untilGreen()` can now see a prior edit agent's changes
mid-run, and two concurrent edit runs on the same repo no longer race each other.

## Added

- **Mid-run staging worktree (#30)**: a run-scoped worktree accumulates each edit-intent agent's
  captured patch, so a standalone `check()`/`gate()` call, a read-only agent, `verify.untilGreen`'s
  first check round, or a later edit agent sees prior edits in THIS run instead of the pristine
  repo. Any non-edit step that needs to see staged content gets its own disposable worktree seeded
  from staging — never a `cwd` pointed directly at the shared staging directory — so a
  side-effecting "read-only" step can't pollute it, two concurrent reads can't race the same
  worktree, and monorepo subdirectory targeting still resolves correctly. `ApplyPatches` (the
  actual apply-to-real-repo step at workflow success) is unaffected — staging is a pure mid-run
  read-side mirror.
- **Advisory repo edit lock (#34)**: a run with any edit-intent agent takes an exclusive lock on
  the repo's canonical root for its duration; a second concurrent edit run on the same repo fails
  fast with a clear error instead of racing git operations against the same worktree. Stale locks
  from a crashed run are reclaimable after 30 minutes via a compare-and-swap takeover.

## Fixed

- A run whose script rejected after some agents already completed used to leak an unrecoverable
  worktree on disk in some cases, and a repo-lock contention failure inside `parallel()`/`pipeline()`
  used to be silently dropped as an ordinary per-item failure instead of halting the run with one
  clear cause. Both fixed as part of an adversarial review of the staging/lock work above.
- Two edit-intent agents making non-conflicting changes to the same file used to fail outright
  (`error: <file>: does not match index`) rather than composing — an accumulation-model bug found
  while testing the conflict-handling case, not the original ask, but severe enough to block
  shipping without it.

## Verification

```bash
node npm/scripts/test.js
go vet ./...
go test ./... -count=1 -race
bash scripts/workflow-verify.sh
pallium workflow audit --run-acceptance
```

Every fix and new mechanism above has a regression test verified by temporarily reverting the fix,
confirming the test fails, then restoring it — including two same-file edit-composition scenarios
run through the real CLI end to end, not just as unit tests.
