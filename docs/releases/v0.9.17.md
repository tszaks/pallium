# v0.9.17

Agent Teams M3, the adoption milestone — complete. This closes the teams trilogy:
M1 made teams possible, M2 made them composable, M3 makes them adopted. Team
templates, decision-contract hardening, external-session attach (the centerpiece),
and an adoption eval extension that measures mid-session decay instead of just
first-call adoption. Four PRs for the milestone itself, plus two more closing real
gaps a verification pass found in M3's own first PR before this release shipped —
one of them a false "already fixed" read that turned out to be a live bug, caught
by testing the claim instead of trusting it.

## Added

- **Team templates.** `pallium team start <goal> --template parallel-review|adversarial-debate`
  spawns a known-good team shape in one call; `team template list|show` browses the
  catalog. Both are built from the exact 3-member team (correctness-reviewer,
  concurrency-reviewer, zombie-hunter) that adversarially reviewed M2's own diff live
  in the previous milestone, generalized off "review Pallium's own code" onto any
  artifact — not theory. Live-proved: 3 real reviewers independently caught the same
  real concurrency bug in a toy repo, each from its own stated lens; the debate pair
  argued with real, opposing evidence pulled from the actual repo under review.
- **External-session attach**, the centerpiece. `pallium team join <team-id> --as <name>`
  registers an already-running agent session — another Claude Code tab, a Codex
  session, a human at a terminal — as a self-driving teammate with no provider
  dispatch at all; the scheduler skips it unconditionally, and it drives itself
  entirely through the ordinary `inbox`/`send`/`tasks claim|complete` CLI. A
  `last_active_at` heartbeat refreshes on every one of those calls (`team status`
  flags it plainly once stale); reading your own inbox is the delivery receipt (no
  turn will ever mark an external member's mail delivered otherwise). Live-proved:
  joined as an external `advisor`, spawned a real claude `worker`, coordinated a
  small real task purely through the mailbox — no clipboard. The worker wrote real
  code, and a reply to advisor stayed undelivered until advisor's own inbox read,
  exactly when the receipt should fire.
- **Decision-contract hardening.** Live M2 dogfooding found real claude/codex
  teammates omitting the decision schema's optional `summary` field on 6 of 6 turns
  in one session — which used to discard the WHOLE decision (messages, claims,
  completions, everything) over one prose field the model itself treated as
  optional. `summary` is no longer required; a decision that still genuinely fails
  schema now notifies the lead's inbox and nudges the member instead of silently
  stranding it behind the scheduler's own "unchanged board" optimization.
- **Turn-prompt claim/complete directive.** A live finding from the external-attach
  proof: a real teammate did the exact work an open task described (a genuine,
  verified code change) but never set `claim_task_id`/`complete_task_id` — the
  board stayed "pending" despite the work landing on disk. The turn prompt now
  says plainly that doing the work isn't enough by itself when an open task
  exists; re-running the exact same live scenario now shows the task correctly
  completed.
- **Adoption eval: team + loop scenarios, mid-session decay.** The eval never
  covered 2 of Pallium's 6 services before this release. `run.sh` now also walks
  every tool call in a transcript in order and flags pallium used in the first
  half but never again in the second — the exact "one preflight call then
  silently reverted to manual" pattern the ledger's own adoption-decay incident
  described, verified against 5 synthetic transcripts before trusting it against
  anything real.

## Fixed

- **A genuinely still-live bug, found by testing a claim instead of trusting it.**
  A verification pass flagged a nudge-erasure finding on M3's first PR as
  "already fixed" by an existing guard. That guard closes a different gap
  (a lost turn-acquisition race); it does nothing for the actual finding.
  Confirmed empirically with a real scheduler run before writing any fix:
  `RunTeam`'s post-turn cleanup was unconditionally erasing the exact nudge
  the decision-contract fix sets mid-turn, undoing it in the same round it
  took effect. Fixed with a CAS-style conditional clear keyed on the nudge's
  value from before the turn ran — same shape as the lease check
  `BeginMemberTurn`/`FinishMemberTurn` already use.
- **`enum` was declared but never enforced anywhere in the schema validator.**
  The team decision schema's `status` field has always declared
  `enum:["active","idle","blocked"]`; nothing in `validateSchemaValue` ever
  read the `enum` keyword at all, so any other string silently passed. Fixed
  generically — checked for any schema node, not a status-only special case,
  so every schema'd `agent()`/`gate()` call benefits. Checked the whole
  codebase for other declared enums first: this was the only one, and
  enforcing it only routes a rare invalid value into the already-tested
  malformed-decision safety path instead of persisting garbage — a strict
  improvement, not a behavior change worth gating on.
- **The real adoption trigger has never mentioned team or loop.** `cmd/agents.go`'s
  `agentsBlock` — the actual text `pallium agents install` writes into real
  `AGENTS.md`/`CLAUDE.md` files — only ever mentioned workflows. Team and loop
  shipped whole milestones ago with zero update here, so a fresh agent had no
  textual path to discovering 2 of Pallium's 6 services. Fixed directly, with
  a regression test (`TestAgentsBlockMentionsEveryService`) so a 7th service
  can't silently repeat this gap.
- **`run.sh`'s cleanup could silently drop the rest of the eval suite.** A
  single flaky `rm -rf` (a lingering subprocess racing the delete — happened
  live, twice) could, under the script's `set -euo pipefail`, abort the ENTIRE
  eval run after whichever task happened to lose that race, with every task
  after it simply never running and no error surfaced beyond the rm noise.
  Now retries briefly and warns instead of losing the rest of the suite over
  one directory the OS's own tmp cleanup reaps anyway.

## Adoption eval — fresh, honest re-run

20 agents, 0 failures. Both brand-new scenarios scored a clean 100 — the team
worker correctly picked the `adversarial-debate` template; the loop worker's
every concrete CLI detail verified against the real binary. The 5 original
`should_use=true` tasks moved to 64.6 recall from the prior 79.8 snapshot;
documented in `eval/adoption/BASELINE.md` with the real per-task judge
reasoning and three compounding, non-isolable reasons a single run can't
cleanly attribute the delta (fresh LLM samples, the trigger block getting
longer mid-measurement, same-model judge/worker — not new, already flagged
in the prior baseline). Presented as a second honest single-run snapshot,
not a regression claim.

**Mid-session decay's own real baseline number is explicitly not obtained.**
`run.sh`'s isolated-HOME design breaks `claude -p` login on at least one real
setup — session auth isn't in the files an isolated HOME can carry, likely OS
keychain. Documented as a known limitation rather than shipping a fake "0%
decay" result from every task silently failing to authenticate, which is
exactly what the first two attempts produced before the failure was traced.
